<objective>
Implement the Community Request and Preferred Comms Config object migrators — the last two objects in the dependency chain. Community Request has a self-referential Parent_Question__c field and a cross-object lookup to Request Flow. Preferred Comms Config is the simplest object with just RecordType mapping.

Purpose: Community Request is the second self-referential pattern (validates the two-pass approach generalizes). The cross-object lookup to Request Flow requires the target's Request Flow Name->Id map, which is built by querying the target after Request Flows are inserted. Preferred Comms is a straightforward insert-with-remap to round out the four objects.

Output: migrate/objects/community_request.py, migrate/objects/preferred_comms.py with tests
</objective>

  <name>Task 1: Community Request migrator with self-ref and cross-object lookup</name>
  <files>migrate/objects/community_request.py, tests/test_community_request.py</files>
  <behavior>
    - migrate_community_requests(source_client, target_client) extracts CFSuite__Data_Settings__c records from source
    - RecordTypeId is remapped from source to target by DeveloperName
    - Records already in target (matched by Name) are skipped
    - Self-referential field CFSuite__Parent_Question__c is resolved via two-pass insert (Process records have no parent, Question records reference Process, Response records reference Question)
    - Cross-object lookup CFSuite__Request_Flow__c is resolved by querying target for Request Flow Name->Id map and replacing source Id with target Id
    - Returns dict with extracted, skipped, inserted counts
    - When a cross-object lookup value has no match in target, that field is set to None (graceful degradation rather than hard failure)
  </behavior>
  <action>
    Create tests/test_community_request.py testing:
    1. Normal migration with RecordType remap and skip: 3 source records, 1 existing in target -> 2 inserted with target RecordTypeId
    2. Self-referential resolution: a Question record's Parent_Question__c points to a Process record -> after two-pass insert, lookup resolves to new Process Id in target
    3. Cross-object lookup: source record has CFSuite__Request_Flow__c = "src-rf-001" -> resolved to target Request Flow Id by Name match
    4. Cross-object lookup miss: source record references a Request Flow not in target -> field set to None (not error)
    5. Empty source: returns all zeros

    Mock sf_api and etl functions.

    Then create migrate/objects/community_request.py:
    - SOBJECT = "CFSuite__Data_Settings__c"
    - FIELDS = ["Name", "RecordTypeId", "CFSuite__Parent_Question__c", "CFSuite__Request_Flow__c", "CFSuite__Active__c", "CFSuite__Description__c", "CFSuite__Order__c", "CFSuite__Question_Text__c", "CFSuite__Response_Value__c"]
    - SELF_REF_FIELD = "CFSuite__Parent_Question__c"
    - CROSS_REF_FIELD = "CFSuite__Request_Flow__c"
    - migrate_community_requests(source_client, target_client) -> dict:
      1. records = etl.extract_records(source_client, SOBJECT, FIELDS)
      2. If empty, return early
      3. Remap RecordTypes (get source/target maps, call etl.remap_record_types)
      4. Resolve cross-object lookup:
         a. Collect unique source Request Flow Ids from records
         b. Query source for those Ids to get their Names: sf_api.query_all(source_client, "SELECT Id, Name FROM CFSuite__Request_Flow__c WHERE Id IN (...)")
         c. Build source_rf_id_to_name map
         d. Query target for Request Flow Name->Id: sf_api.query_all(target_client, "SELECT Id, Name FROM CFSuite__Request_Flow__c")
         e. Build target_rf_name_to_id map
         f. For each record, replace CROSS_REF_FIELD: source Id -> source Name -> target Id (or None if not found)
      5. existing = etl.find_existing_keys(target_client, SOBJECT, "Name", ...)
      6. to_insert = [r for r in records if r["Name"] not in existing]
      7. If to_insert empty, return
      8. Two-pass insert for Parent_Question__c:
         a. Save original parent refs
         b. Null out SELF_REF_FIELD in all records
         c. Insert via sf_api.insert_records
         d. Build name->new_id map
         e. Update records that had non-null parent with resolved new parent Id
      9. Return counts
  </action>
  <verify>
    <automated>cd "/Users/rishigorle/Documents/Ennovative/Sandbox Setup Script" && uv run pytest tests/test_community_request.py -x -v</automated>
  </verify>
