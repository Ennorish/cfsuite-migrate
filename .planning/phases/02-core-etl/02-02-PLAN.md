  <name>Task 1: Entitlement migrator</name>
  <files>migrate/objects/__init__.py, migrate/objects/entitlement.py, tests/test_entitlement.py</files>
  <behavior>
    - migrate_entitlements(source_client, target_client) extracts Entitlement records from source with fields: Name, AccountId, StartDate, EndDate, Status, Type
    - Skips records where Name already exists in target (calls find_existing_keys with key_field="Name")
    - Inserts remaining records into target via sf_api.insert_records
    - Returns dict with keys: extracted (int), skipped (int), inserted (int)
    - When all records already exist, inserted=0 and no insert call is made
  </behavior>
  <action>
    Create migrate/objects/__init__.py (empty package marker).

    Create tests/test_entitlement.py testing:
    1. Normal migration: 3 source records, 1 already in target -> extracts 3, skips 1, inserts 2
    2. All skipped: all source records exist in target -> no insert call made, inserted=0
    3. Empty source: no records in source -> returns extracted=0, skipped=0, inserted=0

    Mock sf_api and etl functions — no real Salesforce calls.

    Then create migrate/objects/entitlement.py:
    - import migrate.sf_api as sf_api and import migrate.etl as etl
    - migrate_entitlements(source_client, target_client) -> dict:
      1. fields = ["Name", "AccountId", "StartDate", "EndDate", "Status", "Type"]
      2. records = etl.extract_records(source_client, "Entitlement", fields)
      3. If no records, return early with all zeros
      4. existing = etl.find_existing_keys(target_client, "Entitlement", "Name", [r["Name"] for r in records])
      5. to_insert = [r for r in records if r["Name"] not in existing]
      6. If to_insert is not empty, sf_api.insert_records(target_client, "Entitlement", to_insert)
      7. Return {"extracted": len(records), "skipped": len(existing), "inserted": len(to_insert)}

    Note: Entitlements do not have RecordTypes in the CFSuite managed package, so no remap_record_types call needed. AccountId lookup resolution is handled by the fact that both orgs share the same Account names in CFSuite sandbox setup — the AccountId field is included as-is. If this assumption breaks, it will be caught during Phase 3 validation.

<task type="auto" tdd="true">
  <name>Task 2: Request Flow migrator with two-pass self-referential insert</name>
  <files>migrate/objects/request_flow.py, tests/test_request_flow.py</files>
  <behavior>
    - migrate_request_flows(source_client, target_client) extracts CFSuite__Request_Flow__c records
    - RecordTypeId is remapped from source to target by DeveloperName
    - Records already in target (matched by Name) are skipped
    - Self-referential fields (CFSuite__Display_Category__c, CFSuite__Category_Journey__c) are resolved via two-pass insert
    - Pass 1: insert all records with self-ref fields set to None
    - Pass 2: update records that had non-null self-ref values with the new target IDs
    - Returns dict with extracted, skipped, inserted counts
  </behavior>
  <action>
    Create tests/test_request_flow.py testing:
    1. Normal migration with RecordType remap: source records have RecordTypeId "src-rt-001", target has different Id for same DeveloperName -> records inserted with target RecordTypeId
    2. Self-referential resolution: a Category Journey record references a Display Category record via CFSuite__Display_Category__c -> after two-pass, the lookup resolves to the new target Display Category Id
    3. Skip existing: records with Name already in target are excluded before insert
    4. Empty source: returns all zeros

    Mock sf_api and etl functions throughout.

    Then create migrate/objects/request_flow.py:
    - import migrate.sf_api as sf_api and import migrate.etl as etl
    - FIELDS = ["Name", "RecordTypeId", "CFSuite__Display_Category__c", "CFSuite__Category_Journey__c", "CFSuite__Active__c", "CFSuite__Description__c", "CFSuite__Order__c", "CFSuite__Entitlement_Name__c"]
    - SELF_REF_FIELDS = ["CFSuite__Display_Category__c", "CFSuite__Category_Journey__c"]
    - SOBJECT = "CFSuite__Request_Flow__c"
    - migrate_request_flows(source_client, target_client) -> dict:
      1. records = etl.extract_records(source_client, SOBJECT, FIELDS)
      2. If empty, return early
      3. source_rt_map = sf_api.get_record_type_map(source_client, SOBJECT)
      4. target_rt_map = sf_api.get_record_type_map(target_client, SOBJECT)
      5. etl.remap_record_types(records, source_rt_map, target_rt_map)
      6. existing = etl.find_existing_keys(target_client, SOBJECT, "Name", [r["Name"] for r in records])
      7. to_insert = [r for r in records if r["Name"] not in existing]
      8. If to_insert empty, return with inserted=0
      9. For each self_ref_field in SELF_REF_FIELDS: call etl.two_pass_insert(target_client, SOBJECT, to_insert, self_ref_field, "Name")
         Actually — two_pass_insert handles ONE self-ref field at a time. But Request Flow has TWO self-ref fields. The approach: do a single pass-1 insert (all self-refs nulled), then do pass-2 updates for each self-ref field separately.
         Revised approach: Create a custom insert flow:
         a. Save original self-ref values: orig_refs = [{f: r.get(f) for f in SELF_REF_FIELDS} for r in to_insert]
         b. Null out all self-ref fields in to_insert
         c. results = sf_api.insert_records(target_client, SOBJECT, to_insert)
         d. Build name->new_id map from results + to_insert Name field
         e. For each record that had non-null self-ref values, update with resolved IDs
      10. Return {"extracted": len(records), "skipped": len(existing), "inserted": len(to_insert)}

    Important: The two self-referential fields (Display_Category__c and Category_Journey__c) both point to other Request Flow records. The hierarchy is: Display Category (root) <- Category Journey <- Case Assignment. A Category Journey's Display_Category__c points to a Display Category record. A Case Assignment's Category_Journey__c points to a Category Journey. So pass-1 inserts all with self-refs nulled, then pass-2 resolves both fields using the name->new_id map.
  </action>
